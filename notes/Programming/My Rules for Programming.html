My Rules for Programming

In the beginning you always want Results. In the end all you want is controll.

This post is a collection of rules of programming that I made for myself by watching Eskil Steenberg talk "How I program in C"

I heighly reccommend you watch the talk before reading the my rules, and making your own rules insted of blindly copying other rules.

The talk consiste of rules that Eskil applys to help him write better programes. Although Eskil made these rules for his C development, I'm going to use following rules for every language that I make. But some rules will be modify to better suit my language need. Because every language has it's own structure of doing things.

As Eskil said the goles of these rules is make our program code more maintable and consitent. There are some other goals but these two I think are most important for good programmer.

Note some rules are my own and have no relation to anything that Eskil said.

Rules

Makde technology footprint small

  • Use the technology that has proven itself.
  • The less things technology does is better.
  • Less dependencie is good. You should not trust external world to your program to function.
  • Good programmer have 100% control to his program.
  • Make program that is going to last forever and ever.

Simple is good

  • Having more code for simplicity is better.
  • Having 10 lines of code doen't have bug in is better then having 2 lines of that does have bug in it.
  • So typing is not the problem.
  • A programmer spend more time reading then writing.
  • Explicit code is better then Implicit code.
  • What you belive your code and what complier belive your code should be the same.

Clever is evil

  • Clever code is makes programmer think that they are doing something very smart, when they are doing something very dumb.
  • Undefine behaiver in program should be avoided.

Crashes are good.

  • Crashes while is devlopment is good.
  • Crashes make you fix thing.
  • Debuggers are super cool if haven't used one.
  • Language shouldn't be complix, it sould be simple as possible.
  • If want to do something complix, you build tool that does that.

Wide code is good code

  • Big names are good because they say explicitly what the program does.

  • Heres how I define things:

    .c 
    // Mutable Global Varibale are all capital & with underscore. 
GLOBAL_VARIBALE
    
// Vriable & Function first letter are capital without undersore or spacing between words.

// Types are all small with underscore between words.
my_type
    
// Spaces are good, it makes thing easer to search.
a = b
a == b
