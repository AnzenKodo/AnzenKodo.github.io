Consensus Algorithm

Solo

  • Type: Single ordering node.
  • Function: Transactions are ordered by this single node sequentially
  • Fault Tolerance: None
  • Use Case: Primarily for development and testing environments

Kafka

  • Type: Distributed ordering service based on Apache Kafka.
  • Fault Tolerance: Crash fault-tolerant (CFT) as long as a majority of nodes are operational.
  • Use Case: Suitable for production networks requiring higher throughput and fault tolerance compared to Solo.
  • Note: Deprecated in favor of Raft.¬†

Raft

  • Type: Crash fault-tolerant (CFT) ordering service based on the Raft consensus protocol.
  • Function: Employs a leader-follower model where a leader is elected among the ordering nodes per channel, and its decisions are replicated to the followers. ¬†
  • Fault Tolerance: Can tolerate the loss of ordering nodes, including the leader, as long as a majority of nodes remain.
  • Use Case: Recommended for production networks.

Forking in Blockchain

Fork happens when a blockchain splits into two or more separate chain.

  • Accidental Fork
    • This fork happens due to network issues, like when two miners find a block at same time.
    • The network resolves this by choosing the longest chain, other chain is discarded.
  • Intentional Fork

It is also called permission blockchain, it is a blockchain network that requires authorization to participate.

Features

  • Permissioned Access
  • Centralized Control

  • Faster Transaction Speed.

Drawbacks

  • Centralization
  • Trust Requirement
  • Limited Transparency: Because not everyone can see the transactions like public blockchain.
  • Lower Security: Compared to public blockchains a private blockchain has fewer nodes.

Examples

  • Hyperledger Fabric
  • Private Ethereum Blockchain: Ulike public Ethereum mainnet, a private blockchain restricts access to select group.
  • R3 Corda: Specifically designed for businesses, particularly in financial services.

Block in Blockchain

Block is data structure that stores a collection of transactions and metadata (block header), forming a sequential chain.

Structure of a Block

  • Block Header
  • Transaction Data
  • Other Data

Block Header Hash:

  • A unique cryptographic hash (e.g., SHA-256) of the block header.
  • Acts as the block's identifier.
  • Ensures integrity; any change in the block alters the hash.

Block Height:

  • The position of a block in the blockchain, starting from 0 (genesis block).
  • Example: The 100th block has a block height of 99.

Linking Blocks in the Blockchain:

  • Blocks are linked in a chain via the Previous Block hash in the block header.

Nonce

Full form Number Only Used Once.

Difficulty Level

  • It is measure of how hard it is to find a new block.
  • Higher Difficulty means that miners need to perform more hashing operations.
  • Target Hash is 256bit's
  • Formula for calculating difficulty:

$$ Difficulty = {\text{Current Target} \over \text{Maximum Target}} $$

  • Current Target is adjusted by the Bitcoin network.

CAP Theorem

The CAP Theorem, also known as Brewer's Theorem.

It states that in distributed store, it is impossible to simultaneously guarantee all three.

  • Consistency: Every read receives the most recent write or error.
  • Availability: Every request receives a response, without guarantee that it contains the most recent write.
  • Partition Tolerance: System continues to operate despite an arbitrary number of messages being dropped or delayed by the network.

Byzantine Generals Problem

How Consensus Mechanisms Solve Byzantine General Problem

  1. Proof of Work (PoW)
    • Mechanism: Miners solve cryptographic puzzles to validate blocks.
    • BFT: Attackers need >51% of computational power to disrupt consensus.
  2. Proof of Stake (PoS)
    • Mechanism: Validators are chosen based on their staked tokens.
    • BFT: Malicious validators risk losing their stake ("slashing").

EOA and Contract Addresses

Externally Owned Accounts (EOAs) in Ethereum are akin to traditional bank accounts. They are controlled by a private key, giving the owner the ability to initiate transactions.

Contract Addresses are unique identifiers for smart contracts deployed on the Ethereum blockchain. Unlike EOAs controlled by private keys, contract accounts are governed by their associated code.
